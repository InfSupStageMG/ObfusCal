# 4. Solution Strategy

This chapter summarises the architectural decisions that realize the quality goals defined in Chapter 1 under the
constraints described in Chapter 2.

## Core Approach

The solution is built around four principles that directly address the problem constraints:

**Privacy by structure, not configuration.** Raw calendar events are never written to the database. They exist only as
in-memory objects for the duration of a single pipeline execution. What gets stored and transmitted is exclusively the
output of the obfuscation pipeline; a `BusySlot` with a minimum of a time range and a maximum of what the user has
chosen to share. A database breach exposes nothing more than anonymous time intervals.

**Federation, not centralisation.** Each organisation runs a sovereign instance. No global authority exists.
Cross-domain exchange happens peer-to-peer through a documented REST API. Any organisation can participate without
establishing a relationship with a central party.

**Pluggable adapters, stable core.** Calendar sources, obfuscation transformers, and storage backends are all hidden
behind interfaces in `ObfusCal.Application` and `ObfusCal.Domain`. Concrete implementations live in
`ObfusCal.Infrastructure` and are resolved via dependency injection. New adapters can be delivered as plugin DLLs
scanned from a `plugins/` folder at startup, with no changes to the core codebase required.

**Asymmetric Operation (Single-Player Value).** The system does not require other organizations to adopt ObfusCal to be
useful. If a client organization cannot or will not run a peer instance, the system falls back to an asymmetric model:

1. **Inbound:** The system ingests the consultant's client commitments via a read-only iCal (`.ics`) link generated by
   the client's existing calendar software.
2. **Outbound:** External parties can view the consultant's merged availability via a secure, public-facing booking link
   or by subscribing to an obfuscated iCal feed generated by the Info Support instance.

This ensures the system provides immediate value to Info Support consultants regardless of external market adoption.

## Quality Goals Addressed by Architecture

| Quality Goal     | Architectural Response                                                                           |
|------------------|--------------------------------------------------------------------------------------------------|
| Privacy          | Obfuscation is a gating condition for storage; raw events are never persisted                    |
| Security         | Entra ID SSO for humans; known-peer validation for machine pushes; strict per-owner data scoping |
| Maintainability  | Interface-based design; pluggable adapters; ADRs for all significant decisions                   |
| Self-hostability | Single `docker compose up`; no external cloud runtime dependency                                 |
| Resilience       | Failed peer syncs are logged and retried; one peer failure does not block others                 |
